## Context

目前，系统已具备社团、推文和制品的爬取、解析和持久化存储功能。随着本地数据体量不断扩充，我们需要系统支持对 `circles` 等表的数据进行多维度统计（题材大盘、周六/周日分流、场馆集聚）。同时，为了实现“题材与受众关系研究”，我们需要设计并在代码中引入合理的“大众受众热度基线”（以二次元大盘各题材的估算占比为参考），从而计算出各题材的“同人偏离度指数 (DBI)”。

## Goals / Non-Goals

**Goals:**
- 在 `src/analytics.py` 中实现题材统计和偏离度（DBI）计算的算法逻辑。
- 预置一份具有代表性的“大众二次元受众热度基线映射表（以各大 IP 的泛受众基数为参考）”。
- 输出格式化为特定结构的 JSON 报告，便于未来对接前端图表。
- 在 `main.py` 中引入 `--analyze-genres <output_path>` 选项。

**Non-Goals:**
- 不在此阶段实现复杂的 Web 页面，仅输出结构化 JSON 文件。
- 偏离度基线数据不依赖实时外部 API 网络抓取（如实时调用 Google Trends，以保证离线可用性与稳定性），采用静态配置与预加载结合。

## Decisions

### 1. 外部受众热度基线的实现方案
- **设计决策**：在代码/配置中内置静态的 `AUDIENCE_POPULARITY_BASELINE`（包含主要同人题材的大众受众占比权重）。若题材不在预置表中，则设置默认的基准分值，使 DBI 默认为 `1.0`。
- **原因**：离线计算是 CLI 的基本原则。通过静态基准，我们既能实现具有学术参考意义的“同人活跃度”分析，又避免了复杂的爬虫开发与动态 API 认证、超时带来的不稳定。

### 2. 题材与时间的映射处理（`土` 与 `日`）
- **设计决策**：在 SQL 统计和输出的 JSON 中，将星期缩写对齐转化为用户更容易理解的 `Day1 (Saturday)` 和 `Day2 (Sunday)`。
- **原因**：这能确保最终产出的分析报告对国际化/中文用户具有更强的可读性，不必在前端再次做复杂的转换。

### 3. 数据层与业务层解耦
- **设计决策**：新建 `src/analytics.py` 专门负责分析业务，而不是将聚合 SQL 塞入 `src/db.py`。
- **原因**：这有利于维护数据库基础读写库的原子性，防止 `db.py` 变得过于庞杂。

## Risks / Trade-offs

- **[Risk] 部分冷门题材因样本量过低导致 DBI 指数失真** → **[Mitigation]** 在统计和计算 DBI 时，对样本数低于阈值（如小于 10 个社团）的题材进行过滤，或者不计算 DBI 归入长尾分类，防止噪点干扰。
- **[Risk] 输出文件所在目录不存在** → **[Mitigation]** 在写入 JSON 前，使用 `os.makedirs(exist_ok=True)` 自动创建父级目录，提升命令行工具的容错能力。
