## Context

在品书商品提取系统中，若将识别引擎 provider 设置为 `cmd`，系统目前是直接调用配置中的外部 CLI 工具（如 `agy`），并把原图路径（可能为数兆大小的 PNG/JPEG，分辨率达到 3000px 以上）替换入 `{image_path}` 中去执行。由于很多本地视觉模型在内存、显存和推理时间上对超大图片有较大瓶颈，因此需要一种预压缩机制。而在 `openai` 模式下，系统在 `encode_image_to_jpeg_base64` 里已经有了自动下采样与降质的处理。我们需要在 `cmd` 模式下，为用户提供一个类似的、可选的本地图片临时压缩管道，在调用命令行工具前自动进行预处理，降低处理开销。

## Goals / Non-Goals

**Goals:**
- 在 `config.yaml` 中新增 `compress`（默认 `false`）、`max_size`（默认 `1500`）、`quality`（默认 `85`）等用于命令行识别引擎的图片预压缩配置。
- 修改 `src/config.py` 以便能加载这些新配置，并提供合理的后备默认值，以保障不影响现有的配置兼容性。
- 在 `src/goods_extractor.py` 的 `extract_goods_via_cmd` 中，若 `compress` 为 `true`，系统在运行自定义命令前，通过 `PIL` 在本地（如 `data/images/tmp/`）生成临时压缩 JPEG 图片，并把命令行里的 `{image_path}` 与 `{abs_image_path}` 替换成该临时路径。
- 用 `try...finally` 结构，确保临时图片在外部命令执行完成（无论成功与否）后被安全删除。

**Non-Goals:**
- **不压缩原始归档图片**：下载并保存在 `data/images/{circle_id}/` 目录下的原图文件保持不变。
- **不对 `openai` 引擎行为做出修改**：`openai` 引擎依然采用内存缩放后转 base64 传输的方式，不产生磁盘临时文件。
- **不提供命令行工具（如 `agy`）本身的安装与维护**。

## Decisions

### 1. 采用本地临时图片目录与安全命名规则
* **选择**：在 `data/images/tmp/` 目录下生成临时压缩图片，并附加随机数或时间戳后缀。
* **原因**：该目录本身在 `data/images` 内，与原图的权限与存储挂载环境相同，便于跨平台兼容，同时不散落在工作区根目录。在生成文件时，添加 `tmp_{timestamp}_` 前缀，防止在测试或后续支持并发执行时发生文件名冲突。

### 2. 利用 `try...finally` 进行临时生命周期管理
* **选择**：在命令行调用外层包裹 `try...finally`，确保在 `finally` 块中执行 `os.remove`。
* **原因**：即使外部命令执行超时、崩溃或解析异常抛出，物理文件也能被 100% 清理，避免在用户磁盘残留大量垃圾临时文件。

### 3. 复用已有 Pillow 图像库
* **选择**：复用 Python 已有的 `Pillow (PIL)` 依赖进行图片预处理。
* **原因**：项目目前已引入 `PIL` 进行 OpenAI Base64 编码，无需引入第三方依赖，且 `PIL` 的 `Image.Resampling.LANCZOS` 采样算法能保证压缩后的文字边缘锐利度。

### 4. 引入“越压越大”的熔断回退机制
* **选择**：如果压缩后的临时文件大小大于或等于原始图片大小的 90%，则立即丢弃临时文件，直接回退使用原图路径。
* **原因**：Twitter 下载的小图（例如 200KB~300KB）通常已经被社交平台本身强力压缩。对其重新进行 JPEG 85 编码不仅无法缩减体积，甚至会因为高保真重建原有压缩噪点而导致文件体积变大。当检测到体积没有明显优势时进行熔断回退，可以保证小图的完美清晰度，同时避免低效的临时文件存储。

## Risks / Trade-offs

* **[Risk] 并行运行导致临时文件名冲突**  
  → **[Mitigation]** 临时文件名命名模板包含时间戳与原图哈希值（如 `tmp_{timestamp}_{orig_filename}`），避免多任务处理时的文件名碰撞问题。
* **[Risk] 图片压缩导致部分微小字体的 OCR 精度降低**  
  → **[Mitigation]** 预压缩 `compress` 的默认配置值设为 `false`。仅在用户遇到本地运行显存不足或速度慢等痛点时才显式开启，并且支持通过 `max_size` 与 `quality` 随时调大分辨率；此外，体积熔断回退机制保障了原本就已经经过强压缩的较小图片（如本就小于 300KB 的图片）不会经历二次图像损耗。
* **[Risk] 临时文件删除因系统锁或权限问题失败**  
  → **[Mitigation]** 在 `finally` 块中进行 `os.path.exists` 检测，且用 `try-except` 包裹删除语句，捕获任何 OS 权限报错（如 Windows 上的文件独占锁），防止删除失败导致整个提取主程序崩掉。
