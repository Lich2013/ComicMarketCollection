## Context

当前系统支持同一个社团同步多条品书推文，每条品书推文对应 `catalogs` 表的一行记录，且下载的品书图片关联在该行记录下。经过 LLM 处理提取后，多条推文中的同人制品都会写入 `goods` 表，且这些商品外键关联到各自的 `catalog_id`。
在数据导出（例如 CSV 导出）时，直接对 `goods` 表进行联表查询，会导致同一个社团多张品书中重复的商品被重复导出，造成双重计数；同时如果新版品书微调了商品价格，旧的价格记录也会并存导出。我们需要设计一种方式进行合并和去重。

## Goals / Non-Goals

**Goals:**
- 实现“库中全留存，读时取最新”的策略。在数据库中完整保留社团的所有品书提取历史。
- 仅在数据查询和导出时（Query/Export），按 `(circle_id, name)`（社团ID + 商品名）维度进行动态去重合并。
- 对于同名商品，以最新的发布推文（即 `tweet_id` 最大的品书）中的属性（如价格、类型）为准；对于其他不同名的追加商品，予以并集保留。

**Non-Goals:**
- 不修改 `catalogs` 与 `goods` 表的数据库结构（Schema）。
- 不改变品书提取时的状态转换逻辑（不修改 `status` 字段）。

## Decisions

### 决策 1：在查询层使用 SQL 窗口函数进行动态合并去重（不改变数据库状态）
- **选择**：使用 SQLite 窗口函数 `ROW_NUMBER() OVER (PARTITION BY g.circle_id, g.name ORDER BY CAST(cat.tweet_id AS INTEGER) DESC)` 对商品进行分区排序并筛选 `rn = 1`。
- **对比替代方案**：更新状态法（解法二），即在写入时将旧品书标为 `superseded`，仅对 `processed` 状态的品书导出。
  - 替代方案的缺点：无法处理“追加/补充”场景（即后一条推文只发布了新周边，标记旧品书为 `superseded` 会导致旧品书里的本子全部漏掉）。
- **理由**：查询去重完美支持“追加/补充”与“修正版价格覆盖”的混合场景，且无需维护复杂的状态机，实现成本低，容错性高。

### 决策 2：推文发布时间的最新性判定
- **选择**：使用 `CAST(cat.tweet_id AS INTEGER) DESC` 进行降序排列。
- **理由**：Twitter (X) 的 `tweet_id` 为 19 位数字的 Snowflake ID，其大小顺序与推文发布时间完全一致。因此数值越大代表推文越新。

## Risks / Trade-offs

- **[Risk] SQLite 版本兼容性风险**：旧版 SQLite 可能不支持窗口函数。
  - **Mitigation**：项目使用 Python 3.11 运行环境，自带的 SQLite 版本均在 3.39+ 以上，已全面支持窗口函数。在单元测试中我们将添加对此 SQL 的执行覆盖验证。
- **[Risk] 性能风险**：随着商品数据量增加，窗口函数计算可能变慢。
  - **Mitigation**：由于每次 Comiket 事件的社团商品总数一般小于 100,000 行，且 `catalogs` 的 `tweet_id` 拥有唯一索引，SQLite 执行此复杂查询通常可在数毫秒内完成，性能完全在可控范围内。
