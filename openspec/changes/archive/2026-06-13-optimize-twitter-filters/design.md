## Context

目前 X.com 上创作者发布的推文非常杂乱，如果将所有包含图片的推文都同步下来会导致大量噪音（如日常插画、食物照、日常灌水转发等）。因此必须细化爬虫端的博文过滤规则。

## Goals / Non-Goals

**Goals:**
- 实现推文时间过滤，默认只采纳 2026.06.01 及之后发布的推文。
- 实现原创与自转判断，跳过所有非目标用户本人的外部推文转发。
- 确保品书图文必须配有图片资源。
- 实现智能上下文关键字匹配，动态从数据库中调取社团的展位号和名字进行多维度搜索，结合通用 CM 关键字共同判定。

**Non-Goals:**
- 不涉及图片本身的 LLM 分析过滤（由后续 `extract-goods` 模块处理，此阶段仅做文本和多媒体元数据层面过滤）。

## Decisions

### 1. 时间过滤与提前终止策略
- **Decision**: 在抓取页面推文时，提取 `<time>` 节点的 `datetime` 属性（ISO 格式）。解析为 Python `datetime` 后，若发现某条推文发布时间早于 `2026-06-01`，则立即跳出该社团的抓取循环。
- **Rationale**: 网页推文按发布时间从新到旧排列，遇到早于阈值的推文，意味着更早的推文也必然不符合要求，提前 `break` 可以避免无谓的滚动和 DOM 查询，提升效率。

### 2. 利用推文状态链接判断转发来源
- **Decision**: 提取推文内指向 `/status/` 的链接的 `href`。如果 `href` 中的用户名与目标社团的 `twitter_username` 不匹配（忽略大小写），则判定为转发他人的推文并予以过滤。
- **Rationale**: 这种方案不需要分析复杂的转发头部 DOM 结构，通过 URL 判定绝对准确且极其简单。

### 3. 基于展位信息的动态智能关键字库
- **Decision**: 在调用 `scrape_twitter_profile` 时，不仅传入通用关键字列表，还将当前社团的 `name`, `author`, `hall`, `block`, `space` 构造成动态匹配条件（包含中日繁简体映射，如 `"东"`/`"東"`），拼装到过滤器的正则表达式或字词包含逻辑中。
- **Rationale**: 创作者在 CM 宣发时，极大概率会带上自己的社团名和展位（如 `"東A-45a"`），将这些信息作为动态过滤词，能大幅提升检索精准度。

---

## Risks / Trade-offs

- **[Risk] 部分老推文由于置顶导致时间判断出现偏差**
  - **Mitigation**: 置顶推文（Pinned Tweet）可能发布于 2026-06-01 之前，如果作为第一条推文处理会导致直接 `break`。需要在定位推文列表时，跳过置顶推文或者单独处理，其余普通推文仍应用 `break` 终止策略。
- **[Risk] 展位写法多样导致匹配漏掉**
  - **Mitigation**: 匹配规则应包含不带连字符的格式（如 `A45a`）和带连字符的格式（如 `A-45a`），并对英文字母做大小写不敏感匹配。
